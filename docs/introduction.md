### 1.2 赛题背景（为什么有这个赛题？）

**问题背景：**
- 现代高性能存储服务器（如全闪存存储）面临一个严重问题：NVMe SSD性能太强了，CPU处理I/O都快跟不上了
- 例如：一块高端NVMe SSD可以达到7GB/s的吞吐量，处理这些I/O请求本身就消耗大量CPU资源
- 如果在存储之上还要跑数据库、应用、中间件，CPU资源就非常紧张
- CPU资源竞争导致整体性能下降

**解决方案：**
- 将计算密集型任务（如压缩/解压）卸载到GPU执行
- GPU有大量并行计算核心，非常适合处理压缩这类可并行化的任务
- 这样可以释放CPU资源，让CPU专注于其他业务逻辑

### 1.3 赛题任务（具体要做什么？）

赛题给出了两条技术路线：

#### 路线一：Block层压缩（我们选择的路线）
```
应用程序 → 文件系统(xfs/ext4) → Block层(dm-compress) → 物理设备
                                        ↓
                                    GPU压缩引擎
```

**优点：**
- 上层支持各种文件系统（xfs、ext4等）
- 对应用透明，不需要修改应用代码
- 通用性强

#### 路线二：文件系统层压缩（可选）
```
应用程序 → 文件系统(btrfs/zfs内置压缩) → 物理设备
```

**优点：**
- 内容感知，可能压缩效率更高

**缺点：**
- 锁定特定文件系统
- btrfs/zfs的绝对性能可能不如xfs/ext4

## 二、赛题意义

### 2.1 技术意义

1. **国产化CPU的补充**：国产CPU（如鲲鹏）在计算密集型任务上相对较弱，GPU offload可以弥补这一短板

2. **存储性能提升**：释放CPU资源，让存储服务器可以承载更多业务

3. **技术创新**：探索GPU在存储系统中的应用，具有前瞻性

### 2.2 实际应用价值

- **云存储服务**：降低存储节点CPU成本
- **数据库服务器**：释放CPU给数据库查询处理
- **大数据处理**：提高存储I/O吞吐量

### 7.2 鲲鹏环境配置步骤

#### 步骤1：系统准备
```bash
# 更新系统
sudo apt update && sudo apt upgrade -y

# 安装基础工具
sudo apt install -y build-essential git wget curl
```

#### 步骤2：安装NVIDIA驱动和CUDA
```bash
# 鲲鹏平台使用ARM版本的CUDA
# 添加NVIDIA仓库
wget https://developer.download.nvidia.com/compute/cuda/repos/ubuntu2204/arm64/cuda-ubuntu2204.pin
sudo mv cuda-ubuntu2204.pin /etc/apt/preferences.d/cuda-repository-pin-600

sudo apt-key adv --fetch-keys https://developer.download.nvidia.com/compute/cuda/repos/ubuntu2204/arm64/3bf863cc.pub

sudo add-apt-repository "deb https://developer.download.nvidia.com/compute/cuda/repos/ubuntu2204/arm64/ /"

sudo apt update
sudo apt install -y cuda-toolkit-12-4 cuda-drivers
```

#### 步骤3：安装内核开发环境
```bash
sudo apt install -y linux-headers-$(uname -r)
```

#### 步骤4：克隆项目并编译
```bash
# 克隆项目
git clone https://github.com/ouyangyipeng/GPU-Off-Load.git
cd GPU-Off-Load

# 运行环境搭建脚本
sudo ./scripts/setup-env.sh --all

# 编译
make all
```

#### 步骤5：安装和测试
```bash
# 安装
sudo make install

# 加载内核模块
sudo modprobe dm_compress

# 创建测试设备
sudo dmsetup create test_compress --table \
    "0 `blockdev --getsize /dev/nvme0n1` compress /dev/nvme0n1 65536 lz4"

# 运行测试
./tests/scripts/run-fio-test.sh
```

### 7.3 测试注意事项

1. **确保GPU正常工作**：运行 `nvidia-smi` 检查
2. **检查内核模块**：运行 `lsmod | grep dm_compress`
3. **查看日志**：`dmesg | tail -20` 查看错误信息
4. **性能对比**：分别在x86和鲲鹏平台运行相同测试

## 附录：常见问题

### Q1: 为什么选择dm层压缩而不是文件系统层压缩？
A: dm层压缩更通用，支持所有文件系统，对应用透明。文件系统层压缩虽然可能压缩效率更高，但锁定特定文件系统。

### Q2: GPU压缩一定比CPU快吗？
A: 不一定。对于小数据块，GPU启动开销可能超过收益。GPU压缩的优势在于大批量数据处理。

### Q3: 如何判断是否应该使用GPU压缩？
A: 根据数据块大小和压缩比判断。我们实现了自适应机制，可以根据情况选择CPU或GPU。

### Q4: 项目能在没有GPU的环境运行吗？
A: 可以。代码实现了CPU回退机制，当GPU不可用时自动使用CPU压缩。
